# Implementation Plan

- [x] 1. 推薦データモデルとDBスキーマの構築
- [x] 1.1 推薦結果のデータモデルを定義する
  - 切り抜き推薦1件を表すPydanticモデルを作成する。セグメントID、動画ID、区間（開始・終了時刻）、推薦スコア（1〜10の整数）、話題要約、魅力紹介テキスト、プロンプトバージョンを持つ
  - スコアの値域制約（1以上10以下）をバリデーションで保証する
  - 関連するデータクラス（推薦オプション、推薦結果、動画+推薦のグループ）も合わせて定義する
  - _Requirements: 2.3, 3.2_

- [x] 1.2 推薦キャッシュ用テーブルをDBに追加する
  - 推薦結果を永続保存するためのテーブルを作成する。セグメントIDとプロンプトバージョンの複合ユニーク制約でキャッシュの一意性を保証する
  - 動画ID単位での検索を高速化するインデックスを作成する
  - 既存のDB初期化処理にテーブル作成を組み込む
  - _Requirements: 2.3_

- [x] 2. 最新アーカイブ選定ロジックの実装
- [x] 2.1 チャンネルの最新N件アーカイブを取得するロジックを実装する
  - チャンネルIDを指定し、同期済みアーカイブを配信日時の降順で最新N件（デフォルト3件）取得するDBクエリを実装する
  - 同期済みアーカイブがN件未満の場合、利用可能な全件を返し、実際の件数を結果に含める
  - 同期済みアーカイブが0件の場合、専用のエラー（未同期エラー）を発生させ、同期コマンドの実行を案内するメッセージを含める
  - 選定された各アーカイブのタイトルと配信日時を返却する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5_

- [x] 3. LLMによる切り抜き適性評価の実装
- [x] 3.1 切り抜き適性を評価するLLMプロンプトとStructured Outputs連携を実装する
  - 動画1本分の全セグメント（要約テキスト一覧）をLLMに送り、各セグメントの切り抜き適性を一括評価するロジックを実装する
  - 評価プロンプトに4軸の基準を明示する: 話題の独立性（文脈なしで楽しめるか）、エンタメ性（面白さ・意外性）、情報価値（有用な知識・ノウハウ）、感情的インパクト（共感・感動）
  - LLMの応答をStructured Outputsで推薦モデルのリストとして取得し、推薦スコア（1〜10）・要約・魅力紹介テキストを一括で得る
  - temperature=0で評価の再現性を確保する
  - _Requirements: 2.1, 2.2, 2.3_

- [x] 3.2 推薦結果のキャッシュ読み書きを実装する
  - 動画単位でキャッシュ済みの推薦結果を検索し、現行プロンプトバージョンと一致する場合はキャッシュを返す
  - キャッシュミス時はLLM評価を実行し、結果をDBに保存する
  - プロンプトバージョンが変わった場合は古いキャッシュを無視して再評価する
  - _Requirements: 2.1_

- [x] 3.3 推薦スコアの閾値フィルタリングを実装する
  - 全動画の推薦結果を集約し、指定閾値（デフォルト7）以上のセグメントのみを抽出する
  - 全セグメントが閾値未満の場合、該当なしの旨と閾値を下げて再実行する案内を返す
  - フィルタリング前後の候補数（全件数とフィルタ後件数）を結果に含める
  - _Requirements: 2.4, 2.5, 2.6_

- [x] 4. 推薦結果フォーマッターの実装
- [x] 4.1 (P) テキスト出力フォーマッターを実装する
  - 推薦結果を動画ごとにグループ化し、各動画のヘッダー（タイトル・配信日時）を表示する
  - 各動画内の推薦候補を時系列順に並べ、話題要約・魅力紹介・推薦スコア・区間（開始〜終了）を表示する
  - 動画グループの表示順は、そのグループ内の最高スコアの降順とする
  - タイムスタンプ付きYouTube URL（`https://www.youtube.com/watch?v=VIDEO_ID&t=START_SECONDS`）を各候補に付与する
  - Task 1のデータモデルのみに依存し、Task 2・3と並行して実装可能
  - _Requirements: 3.1, 3.2, 3.3, 3.4_

- [x] 4.2 (P) JSON出力フォーマッターを実装する
  - 推薦結果をJSON形式で出力するフォーマッターを実装する。動画情報・推薦候補・スコア・区間・URLの全フィールドを含む構造化JSONを出力する
  - Pydanticモデルの直列化機能を活用し、一貫したスキーマで出力する
  - Task 1のデータモデルのみに依存し、Task 2・3と並行して実装可能
  - _Requirements: 3.5_

- [x] 5. CLIサブコマンド統合と動作検証
- [x] 5.1 suggest サブコマンドを定義し、既存CLI体系に統合する
  - `kirinuki suggest <チャンネル指定>` サブコマンドを定義し、既存のCLIエントリーポイントに登録する
  - オプション引数を定義する: `--count`（対象アーカイブ件数、デフォルト3）、`--threshold`（推薦スコア閾値、デフォルト7）、`--json`（JSON出力モード）
  - `--help`で全オプションの説明を表示する
  - _Requirements: 4.1, 4.4, 1.2, 2.5, 3.5_

- [x] 5.2 一連の推薦処理フローを結合し、進捗表示とエラーハンドリングを実装する
  - suggestコマンドから推薦サービスを呼び出し、最新アーカイブ選定→切り抜き適性評価→結果表示の一連処理を実行する
  - 処理の各段階でステータスメッセージを表示する: 対象動画の選定中、セグメント評価中（動画名付き）、結果表示
  - 進捗表示はstderrに出力し、`--json`モード時にstdoutを汚さない
  - 選定された各アーカイブのタイトルと配信日時を処理開始時に表示する
  - 未同期エラー、チャンネル未登録エラー、閾値未満時の案内など、各エラーケースでユーザーフレンドリーなメッセージを表示する
  - LLM API障害時にはキャッシュ済み結果があればそれを返し、なければエラーメッセージを表示する
  - _Requirements: 4.2, 4.3, 1.4, 1.5, 2.6_

- [x] 6. テスト
- [x] 6.1 推薦サービスのユニットテストを実装する
  - 最新N件選定ロジックのテスト: 正常系（3件以上ある場合）、不足時（N件未満）、0件時のエラー発生を検証する
  - 閾値フィルタリングのテスト: 全件通過、一部通過、全件不通過の各ケースを検証する
  - LLM評価のモックを使用し、Structured Outputsレスポンスのパースを検証する
  - キャッシュヒット時にLLM呼び出しがスキップされることを検証する
  - _Requirements: 1.1, 1.3, 1.4, 2.1, 2.4, 2.6_

- [x] 6.2 (P) フォーマッターのユニットテストを実装する
  - テキスト出力の構造テスト: 動画ごとのグループ化、グループ内時系列ソート、グループ間スコア降順を検証する
  - YouTube URL生成テスト: 正常系（任意の秒数）、0秒指定、大きな秒数を検証する
  - JSON出力テスト: 出力がパース可能で全必須フィールドが含まれることを検証する
  - 推薦モデルのバリデーションテスト: スコア範囲外、必須フィールド欠落時のエラーを検証する
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 3.5_

- [x] 6.3 結合テストを実装する
  - モックLLMとテスト用DBを使用し、suggestコマンドのエンドツーエンドフローを検証する
  - テキスト出力モード: URL・スコア・要約・魅力紹介が含まれることを検証する
  - JSON出力モード: JSON出力がパース可能で必須フィールドが存在することを検証する
  - エラーケース: 0件時のエラーメッセージ、閾値未満時の案内メッセージを検証する
  - _Requirements: 4.2, 1.4, 2.6_
